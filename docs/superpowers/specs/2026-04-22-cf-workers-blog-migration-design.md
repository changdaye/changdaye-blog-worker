# Cloudflare Workers 静态博客重写设计

- 日期：2026-04-22
- 目标仓库：`changdaye-blog-worker`
- 旧仓库角色：`changdaye.github.io` 仅作为参考素材与内容来源
- 目标形态：全新 Astro 静态博客 + Cloudflare Workers 静态托管

## 1. 核心目标

本项目不是围绕旧 Jekyll 博客做“迁移管道”，而是：

- 在 `changdaye-blog-worker` 中 **从零重写** 一个新的博客站点
- 页面、组件、样式、内容模型全部按新项目设计
- 旧仓库只作为素材来源，不作为结构继承对象
- 旧文章内容后续 **选择性迁入**，不是先完整搬运再修补
- 图片统一落为本地静态资源，随站点一起部署
- 最终使用 Cloudflare Workers 提供访问入口

## 2. 明确约束

1. **重写优先**：新项目不以兼容 Jekyll 目录结构为目标。
2. **静态优先**：站点仍为静态博客，不做 SSR。
3. **本地资源优先**：图片直接进入项目静态资源目录，不依赖运行时外部图床。
4. **内容选择性迁入**：旧文章按质量、主题、优先级逐步导入，而不是一次性全量平移。
5. **Cloudflare Workers 部署**：最终以 Workers 静态资源托管方式发布。

## 3. 技术方案结论

采用：

- **Astro**：作为新博客框架
- **TypeScript**：作为实现语言
- **Cloudflare Workers + Wrangler**：作为托管与部署入口
- **本地静态资源目录**：承载站点图片与文章图片

不采用：

- Jekyll 延续方案
- 旧模板复刻方案
- 以“imported/_posts”为核心的数据迁移架构
- 运行时 OSS / R2 依赖

建议目标结构：

changdaye-blog-worker/

│   ├── src/
│   │   ├── components/
│   │   ├── layouts/
│   │   ├── pages/

│   │   └── images/

│   ├── package.json

说明：

- `content-source/` 只是临时内容整理区，不是旧仓库镜像
- `scripts/` 只做必要的内容整理辅助，不做“旧站完整迁移系统”

## 5. 新博客信息架构

### 首期页面

1. **首页**
   - 站点标题
   - 个人简介
   - 精选文章列表
   - 最新文章列表

2. **文章详情页**
   - 标题、副标题、日期、标签
   - 正文内容
   - 封面图
   - 目录
   - 上一篇 / 下一篇

3. **标签页**
   - 标签列表
   - 单标签文章列表

4. **About 页面**
   - 按新站风格重写 About
   - 可参考旧 `about.html` 内容，但不直接照搬旧脚本结构

5. **404 页面**

6. **RSS / Sitemap**

### 首期不做

- 旧评论系统
- 旧分享组件
- 旧统计脚本
- 旧 Jekyll 主题兼容层

## 6. 内容模型设计

新内容模型不沿用 Jekyll 的隐式结构，而定义为 Astro 可直接消费的内容集合。

建议字段：

- `title`
- `subtitle`
- `description`
- `date`
- `updated`
- `tags`
- `cover`
- `draft`
- `featured`
- `author`

内容导入原则：

- 旧文章不是先全部导入，而是先挑一批代表性文章验证内容模型
- 旧 frontmatter 只作为参考，不作为必须保留的格式契约
- 导入后的 markdown 应符合新项目标准

## 7. 图片策略

### 7.1 站点图片

例如头像、背景图、封面默认图等：

- 全部进入 `app/public/images/site/`
- 成为新项目自身的一部分

### 7.2 文章图片

- 旧文章里用到的图片，按“实际选入的新文章”逐步本地化
- 保存到 `app/public/images/posts/`
- 文章正文只引用站内静态路径

### 7.3 原则

- 不再以 OSS 在线地址作为最终依赖
- 不再先下载“整个旧站全部图片”再决定是否使用
- 只为新站实际纳入的文章准备图片资源

## 8. 视觉与组件策略

新博客应是重新设计，而不是旧主题翻版。

视觉原则：

- 简洁
- 现代
- 技术博客气质
- 移动端优先
- 强调内容阅读体验

组件原则：

- 小组件拆分
- 布局和内容展示解耦
- 统一主题变量与样式令牌
- 不把旧模板 HTML 当成核心实现基础

## 9. Cloudflare Workers 集成

运行模型：

1. Astro 构建产物输出为静态文件
2. Workers 以静态资源模式托管构建结果
3. Wrangler 负责本地开发与部署

Worker 职责保持最小化：

- 静态文件分发
- 404 fallback
- 基础响应头

## 10. 实施顺序

正确的实施顺序应为：

1. 先搭全新的 Astro 工程
2. 先做新页面骨架、布局、组件、样式系统
3. 先确定新内容模型
4. 再从旧仓库中选择一小批文章导入验证
5. 再把对应图片本地化
6. 最后接入 Cloudflare Workers 部署

不是：

- 先复制旧文章
- 先构造大规模导入脚本
- 先完整本地化旧站全部图片
- 再尝试在其上修修补补

## 11. 风险与边界

### 风险

1. 如果过早批量导入旧内容，重写目标会再次滑回“迁移项目”
2. 如果一次性本地化全部历史图片，仓库会快速膨胀
3. 如果直接沿用旧 HTML/模板片段，组件边界会变差

### 边界

- 新项目首先是一个新博客产品
- 旧项目只是参考库和素材库
- 只有真正进入新站的信息，才值得被导入和本地化

## 12. 结论

`changdaye-blog-worker` 的正确方向是：

- **从零重写一个新的 Astro 静态博客**
- **重新设计页面、组件、样式、内容模型**
- **选择性迁入旧文章**
- **把实际使用到的图片落为本地静态资源**
- **最终部署到 Cloudflare Workers**

这是一个 **rewrite-first** 项目，而不是一个 **import-first migration** 项目。
